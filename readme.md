# Transport Pay System

Единая программно-аппаратная система авторизации и управления транспортными картами. Проект объединяет REST API бэкенд на Go и нативное десктопное клиентское приложение на Kotlin Multiplatform (JVM), обеспечивающее прямой доступ к аппаратному NFC-считывателю.

## 🏗 Архитектура и стек технологий

Проект разделен на изолированные микросервисы (Frontend и Backend), взаимодействующие по HTTP (JSON).

### Frontend (Desktop Client)

Нативное JVM-приложение, разработанное на **Kotlin Multiplatform** с использованием **Compose Multiplatform** для отрисовки интерфейса.

* **Реактивность:** Архитектура построена на паттернах однонаправленного потока данных (UDF). Обновление UI и прослушивание аппаратных прерываний реализовано через `StateFlow` и `SharedFlow` из Kotlin Coroutines.
* **Сеть и Состояние:** Сетевое взаимодействие абстрагировано через Ktor Client. Сохранение пользовательских сессий и токенов (JWT) выполняется в локальном `PlatformStorage`.
* **Многопоточность:** Жесткая изоляция блокирующих I/O операций (опрос COM-порта) в пуле потоков `Dispatchers.IO`, что исключает фризы основного UI-потока Compose.

### Backend (REST API)

Монолитный веб-сервис на **Golang 1.22**, спроектированный по принципам слоистой архитектуры (Clean Architecture).

* **Слои:** Транспортный слой (Handlers на базе фреймворка **Gin**), слой бизнес-логики (Services) и слой доступа к данным (Database/SQLite).
* **Безопасность:** Реализована ролевая модель доступа. Эндпоинты управления терминалами и ключами защищены JWT-авторизацией.
* **Транзакционность:** Списание баланса и запись в лог истории происходят в рамках атомарных SQL-транзакций, исключая рассинхронизацию при гонках данных.

## 🔌 Архитектура аппаратной интеграции (NFC)

В проекте применяется подход **Native Protocol Implementation**. Взаимодействие с NFC-модулем (PN532 V3 + CP2102) реализовано напрямую через USB-UART мост на прикладном уровне JVM, без использования промежуточных C-библиотек (например, `libnfc`) и FFI (Foreign Function Interface).

### Системный уровень (jSerialComm и I/O Kit)

Для связи с мостом CP2102 используется библиотека `jSerialComm`, обращающаяся к POSIX-интерфейсам операционной системы. Решены следующие аппаратные проблемы:

1. **Блокировка портов в macOS:** Стандартные утилиты (включая `libnfc`) пытаются открыть интерфейс `/dev/tty.*`, который блокируется системным демоном в ожидании сигнала DCD (Data Carrier Detect). Приложение использует интерфейс `/dev/cu.*` (Calling Unit), гарантируя неблокирующий монопольный доступ.
2. **Аппаратный контроль потока:** При инициализации применяется конфигурация `termios` с явным отключением Flow Control (`FLOW_CONTROL_DISABLED`). Так как подключены только линии RX и TX, ожидание сигналов CTS/RTS привело бы к переполнению буфера ОС.
3. **Полублокирующее чтение:** Настроен таймаут `TIMEOUT_READ_SEMI_BLOCKING`, блокирующий поток только до поступления первого байта, что исключает пустые циклы процессора.

### Прикладной уровень (High Speed UART Protocol)

Модуль PN532 сконфигурирован аппаратно (DIP-переключатели `I0=0, I1=0`) для работы по протоколу HSU. Взаимодействие реализовано согласно спецификации **NXP UM0701-02**.

Все команды передаются в формате стандартизированного кадра:
`[Preamble (00)] [Start (00 FF)] [LEN] [LCS] [TFI] [DATA] [DCS] [Postamble (00)]`
*(Контрольные суммы `LCS` и `DCS` вычисляются так, чтобы сумма байтов в младшем регистре равнялась `0x00`).*

**Используемые последовательности:**

1. **Пробуждение (WAKE_UP):** `0x55 0x55 0x00 ...`
Байт `0x55` (`01010101` в двоичном виде) генерирует меандр на линии RX, позволяя встроенному блоку Auto-baud чипа PN532 выйти из спящего режима и синхронизировать частоту на 115200 бод.
2. **Конфигурация (SAM_CONFIG):** `00 00 FF 03 FD D4 14 01 17 00`
Команда `0x14` переводит чип из режима эмуляции Virtual Card в рабочий режим считывателя (Normal Mode).
3. **Поиск карты (InListPassiveTarget):** `00 00 FF 04 FC D4 4A 01 00 E1 00`
Команда `0x4A` включает генерацию поля 13.56 МГц и запускает антиколлизионный протокол ISO/IEC 14443-3 (Type A) для считывания MIFARE-меток.

## 🔄 Жизненный цикл транзакции

1. **Hardware:** `NfcService` в фоновой корутине циклично отправляет кадр `InListPassiveTarget`. При обнаружении карты PN532 демодулирует сигнал, упаковывает UID в HSU-кадр (`TFI = 0xD5`) и передает по UART на Mac.
2. **Frontend State:** `PN532Driver` парсит кадр, извлекает UID и транслирует его в `SharedFlow`. Приложение уходит в программный debounce (3 секунды) для защиты от множественных списаний.
3. **Network:** UI Compose реагирует на событие, скрывает модальное окно ожидания карты и асинхронно отправляет HTTP POST запрос (`AuthTransactionRequest`) через Ktor.
4. **Backend Logic:** Go-сервис выполняет SQL-запрос `SELECT ... FROM cards WHERE card_number = ?`, проверяет флаги блокировки и баланс, затем выполняет `UPDATE` баланса и `INSERT` в историю операций, возвращая `200 OK`.

## 🚀 Запуск проекта

### Аппаратное подключение

Подключение адаптера CP2102 к PN532 V3 выполняется перекрестно:

* `5V / 3V3` ➔ `VCC`
* `GND` ➔ `GND`
* `RXD` ➔ `TXD`
* `TXD` ➔ `RXD`

### Локальная разработка (Dev Mode)

**1. Backend (Go)**

```bash
cd backend
go mod download
go run cmd/api/main.go

```

**2. Frontend (Desktop)**
Перед запуском в файле `ApiClient.kt` укажите абсолютный адрес запущенного бэкенда (например, `http://127.0.0.1:8888/api/v1`).

```bash
cd frontend
./gradlew run

```

### Развертывание (Production)

```bash
docker build -t transport-pay-full .
docker run -d -p 8888:8888 --name transport-pay transport-pay-full

```

*Swagger Документация:* `http://localhost:8888/api/v1/swagger/index.html`

## 🛠 Точки роста и технический долг

* **Безопасность (Криптография):** В текущей реализации транзакция подтверждается публичным UID карты (например, `89FC2006`), который подвержен клонированию (Emulation/Proxmark3). Требуется внедрение команд `InDataExchange` для криптографической аутентификации в защищенных секторах MIFARE Classic/DESFire.
* **Автономная работа (Offline Mode):** Приложение синхронно ожидает ответ бэкенда. Для валидаторов в транспорте необходимо реализовать локальную SQLite БД (через Room/SQLDelight) для буферизации транзакций с последующей асинхронной синхронизацией при восстановлении связи.
* **Сборка фреймов:** Заменить захардкоженные байтовые массивы в Kotlin-коде на динамический `FrameBuilder` с автоматическим побайтовым расчетом `LCS`/`DCS` для масштабирования команд.
